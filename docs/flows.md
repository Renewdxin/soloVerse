# Commitment Agent · 用户视角流程与用例

> 配套：[architecture.md](./architecture.md) · [plan.md](./plan.md)。
> 这份不是技术，是"在工作群里真实看到的样子"。**开工前用它确认每个 case 的措辞、节奏、边界。**

---

## 0. 基调：隐形为主

Bot 默认**隐形**——绝大多数时候不出声。它只在四个时刻出现：

1. 有人许下承诺时（确认一句）
2. 每天一次「今日 todo」、每周一次「本周进度」
3. 某人有事临期时（**一天最多一句**温和提醒）
4. 完成时（一句确认）+ 被问到时（答查询）

**没有消息 = 一切正常。** 它不刷屏、不连环催、**不羞辱、不公开施压**（点任务事实可以，评价人不行）。

- `[名字]` = 群里某人；`[Bot]` = bot（**问责在群里公开；个人 digest 走私聊**，见架构原则 7）。

## 1. 主线

### 1.1 捕获 + 群内确认
```
[小李] 这两天把登录的 bug 修一下，周三前提 PR
[Bot] 记一笔 👉 @小李 · 周三(6/3)前 · 修复登录 bug 并提 PR。对吗？（不对回「改」/「不是」）
[小李] 对
[Bot] 好，跟上了。
```

### 1.2 静默进展（默认状态）
> 周二 bot 查到关联分支有提交 → **不出声**。on-track 没消息就是好消息。

### 1.3 临期提醒（一天最多一次，温和、带资料）
```
[Bot] @小李 登录 bug 今天 due，PR 还没见，进度怎么样？
      （顺手）之前那个 issue 可能有用：<link>
[小李] 在弄了，环境有点问题
[Bot] 收到。
```
> 决策点：**一天最多提醒一次**；就事论事、带点资料；对方回了就**收声**，不追问。

### 1.4 完成 + 确认
```
[小李] 好了 PR #123
[Bot] （查 #123 → 已合并）✅ 看到了，这条结了。
```
> 也可能 bot 自己轮询到合并、不等他说就 ✅。

### 1.5 逾期 + 改期 / 放弃
```
[Bot] @小李 登录 bug 过了今天还没完成，还做吗？给个新时间，或回「放弃」。
[小李] 改周五
[Bot] 好，周五(6/5)。
```
> 逾期**不羞辱**，给体面出口。

## 2. 查证的几种形态

- **2.1 GitHub（认证）**：关联 PR/commit/issue，用 token 看到 merged / CI 绿 → 确定性判完成。
- **2.2 任意 link**：`[小王] 设计稿周四前出，figma <link>` → 到点 bot 抓 figma 看进度。
- **2.3 私有链接打不开**：`[Bot] @小王 这个 Notion 我没权限打不开，你直接说做没做我记一下。`（**不假装**看得到）
- **2.4 manual**：没有可查链接（"给客户打电话"）→ 到点 bot 问一句，对方自报。

> **可信度分层**：只有 **2.1 GitHub merged/CI** 这类高可信信号能让 bot **自动判完成**；**2.2 link 抓取**最多说"看到 / 没看到变化"（弱信号，不自动结案）；manual 靠自报；none 只提醒。**宁可 inconclusive，不假装完成。**

## 3. 项目面板（搜索调度，全在群里、全员可见）

### 3.1 每日「今日 todo」
```
[Bot] 📋 今日 todo · 6/3
      • @小李 登录bug修复（今天 due）
      • @小张 接口文档（今天 due）
      • @小王 设计稿（明天 due）
```
> 这是**主要的、被动的问责面**——清单本身就是温和压力，不靠催。

### 3.2 每周「本周进度」
```
[Bot] 📊 本周进度 6/1–6/7
      ✅ 完成 3：登录bug<link> · 设计稿<link> · 客户确认
      ⏳ 逾期 1：@小李（已改周五）
      🚧 零进展 1：@小张 接口文档
```

### 3.3 随时查（群里谁都能问）
```
[老板] @Bot 这周谁还没交
[Bot] 未完成 · @小李 登录bug(改周五) · @小张 接口文档(今天 due)。已完成 · @小王 设计稿 ✅
```

## 4. code review（专业；问题开成 thread，不在群里顺手提）
```
[小李] PR #123 好了
[Bot]（只读 diff，发现 token 过期路径没测试）
     → 直接在 PR #123 上开一条 review thread：「token 过期这条建议补个测试用例覆盖」
     群里最多一句：review 了 #123，提了 1 条（在 PR 上）。
```
> 边界：问题**进 GitHub 的 thread**（PR review 评论 / 开 issue）——正式、可追踪、在该解决的地方解决；**不在群里顺手提一嘴**。bot **能写评论 / 开 issue，但不改代码、不开代码 PR**（§16）。一条没解决的 review thread / issue 本身也能被当成事项跟踪。

## 5. 边界 & 难 case（这些最该你拍板）

| 情况 | bot 的处理 |
|---|---|
| **抽取误判**（把闲聊当承诺） | 有人回「不是」→ 撤回、不计；置信低**宁可不记** |
| **模糊 deadline**（"尽快""这几天"） | 反问「具体哪天？」，不瞎定 |
| **责任人不清** | 问「这条算谁的？」，不默认 |
| **装死 / 已读不回** | **不追加催问、不公开施压、不羞辱**——只是继续留在「今日 todo / 本周进度」的中性清单里（清单点的是任务，不是人） |
| **撞 quiet hours / 周末** | 压到工作时间再说 |
| **同一条想多催** | 一天最多一次，记得说过的不重复 |
| **有人嫌烦** | on-track 本就不出声；可 `/snooze`、可调频率 |

## 6. UX 原则（从 case 提炼）

1. **隐形为主**：默认不出声，只在许诺 / 每日清单 / 临期一次 / 完成 / 被问 时出现。
2. **清单 > 催**：每日 todo + 每周进度是主要问责面；催是辅助，一天最多一次。
3. **点任务、不点人格**：可以公开标记任务归属与状态，但不做道德评价、不连环施压、不羞辱；装死的人留在中性清单里就够了。
4. **公开、可纠正、可改期**：全在群里，说错当场改，逾期给出口。
5. **增值不只施压**：带资料、做**专业 review**（读码只读，但会写评论 / issue——对工作流有副作用，不是纯只读）——但**不改代码**。
6. **不假装**：看不到就说看不到，降级自报。
7. **宁可漏不可误**：误记、冤枉人，比漏一条更伤。**误标完成（没做却说做了）最伤**——只有高可信证据才自动结案。
8. **知情可纠正**：只在已接入的群记事；任何人可纠正 / 删除自己的事项（`/forget`、`/stop tracking me`），bot 判错一句即推翻。
